# OTA升级指南

本指南介绍如何使用OTA功能对边缘设备和Edge Agent进行远程升级。

## OTA升级概述

OTA（Over-The-Air）升级允许远程更新边缘设备上的软件，包括：

- **Edge Agent升级**：Agent程序更新
- **edge_infer升级**：推理框架更新
- **模型更新**：通过模型部署实现（参考《模型部署指南》）
- **配置更新**：通过配置下发实现

## 升级策略

### 灰度发布

逐步扩大升级范围，降低风险：

```
阶段1：1台设备（24小时观察）
阶段2：10%设备（48小时观察）
阶段3：50%设备（72小时观察）
阶段4：全部设备
```

### 分批升级

按设备分组分批升级：

```
批次1：巡检组A（10台）
批次2：巡检组B（15台）
批次3：巡检组C（20台）
```

## 创建升级任务

### 第一步：准备升级包

升级包结构：

```
upgrade_package_v1.0.1.zip
├── edge-agent/           # Edge Agent升级包
│   ├── edge-agent        # 可执行文件
│   ├── config.yaml       # 配置文件
│   └── scripts/          # 升级脚本
├── edge-infer/           # edge_infer升级包（可选）
│   ├── bin/edge_framework
│   └── lib/plugins/
└── manifest.yaml         # 升级清单
```

**manifest.yaml示例**：

```yaml
version: "1.0.1"
type: "agent"
description: "Bug修复和性能优化"
components:
  - name: "edge-agent"
    version: "1.0.1"
    checksum: "sha256:abc123..."
    size: 10485760
  - name: "config"
    version: "1.0.0"
    checksum: "sha256:def456..."
requirements:
  min_agent_version: "1.0.0"
  min_disk_space: 1073741824  # 1GB
release_notes: |
  - 修复MQTT重连问题
  - 优化内存使用
  - 支持新的模型格式
```

### 第二步：上传升级包

1. 点击"OTA管理" → "升级包管理"
2. 点击"上传升级包"
3. 填写信息：

| 字段 | 说明 | 示例 |
|------|------|------|
| 版本号 | 语义化版本 | 1.0.1 |
| 升级类型 | agent/infer/config | agent |
| 升级包 | 上传ZIP文件 | upgrade_v1.0.1.zip |
| 发布说明 | 更新内容 | Bug修复和性能优化 |

4. 上传并验证

### 第三步：创建升级任务

1. 点击"创建升级任务"
2. 配置任务：

**基础配置**：

| 参数 | 说明 | 示例 |
|------|------|------|
| 任务名称 | 任务显示名称 | Agent升级v1.0.1 |
| 升级包 | 选择已上传的包 | v1.0.1 |
| 目标设备 | 选择要升级的设备 | 勾选设备或分组 |

**升级策略**：

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| 升级方式 | 全量/差量 | 差量 |
| 并发数 | 同时升级的设备数 | 5 |
| 失败重试 | 失败后重试次数 | 3 |
| 自动回滚 | 失败后自动回滚 | 是 |

**调度配置**：

| 参数 | 说明 | 选项 |
|------|------|------|
| 执行时间 | 立即/定时 | 立即 |
| 维护窗口 | 允许升级的时间段 | 02:00-06:00 |

3. 点击"创建任务"

## 失败处理

### 自动回滚

当以下情况发生时，自动回滚：

- 校验失败
- 安装失败
- 验证失败
- 服务无法启动

回滚流程：

```
1. 检测到失败
2. 停止新版本服务
3. 恢复备份版本
4. 启动旧版本服务
5. 验证功能正常
6. 记录失败日志
```

### 手动回滚

如果需要手动回滚：

1. 进入"升级任务"详情
2. 选择失败的设备
3. 点击"回滚"
4. 系统恢复到升级前状态

### 失败分析

查看失败原因：

| 失败类型 | 常见原因 | 解决方案 |
|---------|---------|---------|
| 下载失败 | 网络问题 | 检查网络，重试 |
| 校验失败 | 文件损坏 | 重新上传升级包 |
| 安装失败 | 磁盘空间不足 | 清理空间 |
| 启动失败 | 配置错误 | 检查配置文件 |

## 升级最佳实践

### 1. 充分测试

升级前在测试设备上验证：

- 功能测试
- 性能测试
- 兼容性测试

### 2. 准备回滚方案

确保可以快速回滚：

- 保留备份
- 准备回滚脚本
- 文档化回滚流程

### 3. 选择合适时机

在业务低峰期升级：

- 凌晨2-6点
- 周末
- 节假日

### 4. 监控升级过程

实时监控：

- 设备在线率
- 服务运行状态
- 错误日志

### 5. 分阶段升级

按照以下顺序：

```
测试设备 → 小批量 → 中批量 → 全量
```

## 版本管理

### 版本号规范

使用语义化版本：

```
主版本.次版本.修订版

示例：1.2.3
- 主版本(1)：不兼容的API修改
- 次版本(2)：向下兼容的功能性新增
- 修订版(3)：向下兼容的问题修正
```

## 常见问题

### Q1: 升级时间一般多久？

取决于升级包大小和网络速度：
- Agent升级：5-10分钟
- 完整升级：15-30分钟

### Q2: 升级期间服务会中断吗？

灰度升级时只有正在升级的设备会中断，其他设备正常运行。单个设备中断时间约2-5分钟。

### Q3: 如何跳过某个版本？

系统支持跨版本升级，直接从v1.0.0升级到v1.2.0。

### Q4: 升级失败后怎么办？

系统会自动回滚。如果回滚也失败，请联系技术支持。
