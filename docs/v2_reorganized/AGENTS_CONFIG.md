# 智能体配置方案

> 更新时间：2026-03-06  
> 目的：持续更新和设计产品，各个AI智能体对应的子代理管理  
> **策略**：混合模式 - 核心工作持久化，临时任务快速化

---

## 核心策略

### ✅ 持久化Session（核心工作）
**适用场景**：明确的任务角色分工，长期维护的产品工作

**角色列表**：
1. 三大核心智能体专家（感知/认知/行动）
2. 支撑智能体（硬件/软件/文档/测试）

**特点**：
- UI可见（左上角显示）
- 历史完整保留
- 可随时进入继续工作
- 适合持续迭代

---

### ⚡ 临时子代理（快速任务）
**适用场景**：其它任务，未指定任务角色

**任务类型**：
- 一次性分析
- 批量处理
- 快速查询
- 临时翻译

**特点**：
- 用完即销
- 结果直接返回
- 不占用长期资源

---

## 架构设计原则

### 核心思路：混合模式

```
持久化Session（长期维护） + 临时子代理（快速任务）
```

## 三、管理策略

### 3.1 创建持久化Session

**创建时机**：
- 项目启动时创建核心智能体
- 新增专业领域时创建支撑智能体

**创建命令**（示例）：
```
请创建一个"硬件工程师"持久化session，负责妙算3的集成设计
```

### 3.2 维护持久化Session

**定期维护**（每周）：
- 检查各个session的状态
- 清理不再需要的session
- 更新session的任务清单

**进入session工作**：
```
# 方式1：点击UI左上角的session名称
# 方式2：告诉我"进入硬件工程师session，看看PCB设计进展"
```

### 3.3 清理策略

**何时清理**：
- 项目阶段完成（如V1.0发布后）
- 智能体不再需要（如某个专项完成）
- 资源占用过多时

**清理方式**：
```
subagents(action="kill", target="session_key")
```

## 五、判断标准与使用示例

### 🎯 如何判断使用哪种方式？

#### ✅ 使用持久化Session的标准

**满足以下任一条件**：
1. ✅ 任务有**明确的角色归属**（如硬件工程师、软件架构师）
2. ✅ 需要**持续迭代**的工作（如产品设计、架构优化）
3. ✅ 需要**保留历史记录**（如设计决策、讨论过程）
4. ✅ 属于**核心产品开发**（如三大智能体的实现）
5. ✅ 需要**多次访问和更新**（如文档维护）

**示例**：
```
✅ "硬件工程师，优化妙算3的散热设计"
   → 有明确角色，需要持续迭代
   
✅ "软件架构师，更新系统架构图"
   → 有明确角色，需要保留历史
   
✅ "感知智能体专家，继续RCMT训练优化"
   → 核心产品开发，需要多次访问
```

---

#### ⚡ 使用临时子代理的标准

**满足以下所有条件**：
1. ⚡ 任务**未指定特定角色**
2. ⚡ **一次性任务**，不需要长期跟踪
3. ⚡ **快速获取结果**即可
4. ⚡ 不需要保留详细历史

**示例**：
```
⚡ "@delegate 分析这个PR的代码变更"
   → 一次性分析，快速获取结果
   
⚡ "@delegate 生成今天的工作日报"
   → 临时任务，不需要长期保留
   
⚡ "@delegate 比较这两个方案的成本"
   → 一次性对比，快速获取结论
```

## 六、最佳实践

### ✅ 推荐做法

1. **核心工作持久化**
   - 产品设计、架构设计等长期工作
   - 需要多次迭代的任务
   - 需要保留历史的场景

2. **临时任务快速化**
   - 一次性分析
   - 不需要保留的工作
   - 快速获取结果

3. **定期维护**
   - 每周检查session状态
   - 清理不需要的session
   - 更新任务清单

### ❌ 避免做法

1. **不要过度创建持久化session**
   - 会占用资源
   - 增加管理复杂度

2. **不要在持久化session做临时任务**
   - 会污染历史记录
   - 降低查找效率
